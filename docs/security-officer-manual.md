# MCP Gateway 보안 담당자 매뉴얼

**대상**: MCP Gateway 포털 관리자 (보안 담당자)  
**버전**: 1.0.0  
**작성일**: 2026-03-31

---

## 1. 개요

MCP Gateway 포털은 폐쇄망 개발자들이 AI 코딩 도구를 통해 수행한 모든 외부 검색 요청을 모니터링하고, 민감 정보 차단 규칙을 관리하는 보안 담당자 전용 웹 포털입니다.

**포털 접속 URL**: `http://{gateway-host}:13000`

---

## 2. 로그인

### 2.1 접속

1. 브라우저에서 포털 URL 접속
2. Keycloak 로그인 화면으로 자동 리다이렉트
3. 관리자 계정으로 로그인

> **주의**: 포털 계정은 IT 담당자에게 별도 요청하여 발급받습니다.  
> 계정 발급 문의: Keycloak Admin Console (`http://{gateway-host}:18080`)

### 2.2 로그아웃

우측 상단 사용자명 옆 **[로그아웃]** 버튼 클릭

---

## 3. 대시보드

로그인 후 첫 화면. 오늘의 현황을 한눈에 파악할 수 있습니다.

### 3.1 통계 카드

| 카드 | 내용 |
|------|------|
| **오늘 검색 요청** | 당일 전체 검색 요청 건수 |
| **차단 건수** | 당일 차단이 적용된 검색 건수 |
| **활성 차단 규칙** | 현재 활성화된 규칙 수 |

### 3.2 차트

- **서비스 분포 (도넛 차트)**: Context7 vs Exa 서비스별 요청 비율
- **시간대별 요청 추이 (막대 그래프)**: 최근 24시간 시간대별 요청량

### 3.3 확인 포인트

- 차단 건수가 급격히 증가했다면 → **검색 로그** 탭에서 원인 확인
- 특정 시간대에 요청이 집중된다면 → 해당 개발자 IP 확인

---

## 4. 검색 로그 모니터링

**메뉴**: 상단 nav → **검색 로그**

### 4.1 화면 구성

| 컬럼 | 설명 |
|------|------|
| 시간 | 검색 요청 발생 시각 |
| 사용자 (IP) | 요청한 개발자 PC의 IP 주소 |
| 서비스 | Context7 또는 Exa |
| 검색 내용 | 실제 검색어 (원본, 차단 전) |
| 상태 | HTTP 응답 코드 (200 = 정상) |
| 차단 | 차단 적용 시 규칙명 배지 표시 (행 클릭 → 차단된 원본 텍스트 확인) |

### 4.2 필터 기능

- **서비스 선택**: 전체 / Context7 / Exa
- **차단된 항목만**: 체크 시 차단이 발생한 요청만 표시

### 4.3 상세 조회

로그 행을 **클릭**하면 상세 정보가 펼쳐집니다:
- 원본 검색어 전문
- 적용된 차단 규칙명
- 차단된 원본 텍스트 (예: `"10.10.20.30"`, `"900101-1234567"`)
- 요청 경로 (`GET /v2/libs/search`)

### 4.4 이상 징후 확인 방법

**특정 개발자가 반복적으로 민감 정보를 검색에 포함시키는 경우:**
1. 검색 로그에서 해당 IP로 필터
2. 차단 규칙 및 원본 텍스트 확인
3. 필요 시 감사 로그와 대조하여 이력 확보
4. 해당 개발자에게 보안 교육 요청

## 5. 차단 규칙 관리

**메뉴**: 상단 nav → **필터 규칙**

### 5.1 기본 규칙 목록 (13개)

시스템 시작 시 자동으로 로드되는 기본 규칙입니다.

| # | 규칙명 | 유형 | 방향 | 패턴 예시 |
|---|--------|------|------|-----------|
| 1 | 주민등록번호 | 정규식 | 양방향 | `\d{6}-[1-4]\d{6}` |
| 2 | IP 주소 (응답) | 정규식 | 응답 | `\d+\.\d+\.\d+\.\d+` |
| 3 | 민감 키워드 | 키워드 | 양방향 | password, secret 등 |
| 4 | AWS 액세스 키 | 정규식 | 요청 | `AKIA[0-9A-Z]{16}` |
| 5 | 클라우드 시크릿 | 정규식 | 양방향 | `aws_secret=...` 패턴 |
| 6 | 인증 토큰 키워드 | 키워드 | 양방향 | API_KEY, JWT_SECRET 등 |
| 7 | 사설 IP 대역 | 정규식 | 요청 | `10.x.x.x`, `192.168.x.x` 등 |
| 8 | 내부 도메인 | 정규식 | 양방향 | `*.internal`, `*.corp` |
| 9 | DB 접속정보 | 정규식 | 양방향 | `postgres://user:pass@host` |
| 10 | 휴대폰번호 | 정규식 | 양방향 | `010-xxxx-xxxx` |
| 11 | 이메일 주소 | 정규식 | 요청 | `user@domain.com` |
| 12 | 신용카드번호 | 정규식 | 양방향 | 카드번호 패턴 |
| 13 | 계좌번호 | 정규식 | 양방향 | 은행 계좌번호 패턴 |

### 5.2 새 규칙 추가

**[+ 새 규칙]** 버튼 클릭 후:

| 필드 | 설명 | 예시 |
|------|------|------|
| 이름 | 규칙 식별명 | `사내 프로젝트명` |
| 유형 | 정규식 또는 키워드 | `keyword` |
| 패턴 | 매칭 패턴 | `Project-Alpha,PROJ-ALPHA` (키워드는 쉼표 구분) |
| 서비스 | 적용 서비스 | `all` |
| 방향 | 요청/응답/양방향 | `request` |
| 설명 | 규칙 목적 설명 | `사내 프로젝트 코드명 유출 방지` |

> **규칙 추가 전 확인사항**: `docs/filter_rules.md` 참조하여 중복/모순 여부 확인

### 5.3 규칙 활성/비활성

규칙 행의 **[활성]** / **[비활성]** 버튼 클릭으로 즉시 토글.

> 비활성화된 규칙은 차단 처리에 포함되지 않습니다.

### 5.4 규칙 삭제

규칙 행 우측 **[삭제]** 클릭 → 확인 다이얼로그에서 삭제.

> **주의**: 기본 제공 13개 규칙은 재설치 시 자동 복구되지만, 추가한 규칙은 삭제 시 복구 불가.

## 6. 감사 로그

**메뉴**: 상단 nav → **감사 로그**

시스템 내 모든 액션의 이력을 조회합니다.

### 6.1 액션 유형

| 액션 | 배지 색상 | 설명 |
|------|-----------|------|
| 검색 | 회색 | 차단 없이 정상 검색 완료 |
| 차단 | 노란색 | 차단이 적용된 검색 |
| 규칙 생성 | 파란색 | 차단 규칙 추가 |
| 규칙 수정 | 파란색 | 차단 규칙 수정 |
| 규칙 삭제 | 빨간색 | 차단 규칙 삭제 |

### 6.2 필터 기능

- **액션 선택**: 특정 액션 유형만 필터
- **사용자 IP 검색**: 특정 IP 입력 후 [검색] 클릭

### 6.3 상세 컬럼

차단 액션의 경우 상세 컬럼에:
- 차단 건수 (예: `차단 2건`)
- 적용된 규칙명 배지 + 차단된 원본 텍스트 (예: `사설 IP 대역` — `"10.10.20.30"`)
- 여러 규칙이 적용된 경우 각각 별도 행으로 표시

## 8. 주요 업무 시나리오

### 시나리오 A: 일일 보안 점검

1. **대시보드** 접속 → 오늘의 차단 건수 확인
2. 차단 건수가 전일 대비 급증했다면 → **검색 로그** → 차단된 항목만 체크
3. 반복적으로 민감 정보를 입력하는 IP 식별
4. 해당 IP 개발자에게 보안 교육 통보

### 시나리오 B: 신규 프로젝트 착수 시 규칙 추가

1. **필터 규칙** → **[+ 새 규칙]**
2. 프로젝트 코드명, 내부 시스템명 등 추가
3. 유형: `keyword`, 방향: `request` (외부 전송 차단)
4. 활성화 확인 후 저장

### 시나리오 C: 특정 개발자 활동 추적

1. **검색 로그** → 서비스 필터는 전체, 차단된 항목만 체크
2. 특정 IP 행 클릭 → 원본 검색어 확인
3. **감사 로그** → 해당 IP 검색 → 전체 이력 확인

### 시나리오 D: 규칙 오탐(False Positive) 처리

1. **검색 로그**에서 불필요하게 차단된 항목 확인
2. 적용된 규칙명 확인 (행 클릭 → 상세)
3. **필터 규칙** → 해당 규칙 **[비활성]** 처리
4. 규칙 패턴 수정 후 재활성화

## 9. 긴급 대응

### 심각한 민감 정보 유출 시도 탐지

1. **검색 로그**에서 원본 텍스트 캡처
2. 해당 IP 식별
3. 네트워크 담당자에게 해당 IP 트래픽 차단 요청 (방화벽 수준)
4. 보안 감사 보고서 작성 (감사 로그 내보내기 활용)

## 10. 개발자 PC 보안 파일 관리

### 개요 (3계층 방어)

개발자 PC의 보안 시스템은 3계층으로 구성됩니다:

| 계층 | 파일 | 위치 | 역할 | 트리거 |
|------|------|------|------|--------|
| **-1차 훅** | `search-guard-hook.js` | `.opencode/hooks/` | 정규식 민감정보 차단/치환 | 자동 (MCP 호출 시) |
| **-1차 훅** | `search-log-hook.js` | `.opencode/hooks/` | 검색 로컬 감사 로깅 | 자동 (MCP 호출 후) |
| **-1차 설정** | `hooks.json` | `.opencode/hooks/` | 훅 바인딩 (context7/exa) | 자동 |
| **-1차 설정** | `settings.json` | `.claude/` | oh-my-openagent 훅 설정 | 자동 |
| **0차 정책** | `AGENTS.md` | 프로젝트 루트 | 보안 정책 (instructions) | 자동 (항상 로딩) |
| **0차 정책** | `security-policy.md` | `.opencode/rules/` | 보안 정책 룰 | 자동 (instructions) |
| **0차 에이전트** | `search-guard.md` | `.opencode/prompts/agents/` | LLM 의미 기반 검토 | 수동 (/search-guard) |

> ℹ️ `.claude/settings.json`은 oh-my-openagent 훅 설정 파일입니다. Claude 서비스와 무관합니다.

### 보안 파일 수정 방법

1. 보안 패키지 git 저장소 수정
   ```
   git clone https://internal-git/security/opencode-security-agent.git
   ```
2. 해당 파일 수정
3. 검토 후 push → 개발자 PC에서 git pull 시 자동 반영

### 훅 패턴 추가 시

`search-guard-hook.js`의 `DENY_PATTERNS` 또는 `REPLACE_PATTERNS` 배열에 추가:

```javascript
// 차단 패턴 추가 (검색 거부)
{ name: '새로운PII', pattern: /새로운패턴/g },

// 치환 패턴 추가 (일반화 후 검색)
{ name: '새패턴', pattern: /새로운패턴/g, replacement: '[일반화]' },
```

### 새 MCP 서비스 추가 시

1. `hooks.json`과 `.claude/settings.json`에 새 matcher 추가:
   ```json
   { "matcher": "mcp__{new-service}__*", "hooks": [...] }
   ```
2. Gateway backend에 `/proxy/{service}/*` 라우트 추가
3. 배포 패키지 업데이트 → 개발자 PC 재배포
